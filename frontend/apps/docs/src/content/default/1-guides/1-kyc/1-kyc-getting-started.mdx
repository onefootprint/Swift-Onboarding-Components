## Add Footprint.js to your app

1. Go to the Footprint [Playbooks tab](https://dashboard.onefootprint.com/playbooks) and create a new Playbook, selecting 'Onboard people'.

   - On the "Your Playbook" screen, click the edit button to change the fields you plan to collect.
   - On the "Authorized scopes" screen, select the fields you require access to decrypt.

2. Grab the Onboarding Publishable Key, for example `pb_test_VMooXd04EUlnu3AvMYKjMW`.

3. Add the `footprint.js` script to your app.

```html
<head>
  <link
    rel="stylesheet"
    href="https://unpkg.com/@onefootprint/footprint-js@latest/dist/footprint-js.css"
  />
</head>
<body>
  ...
  <script
    crossorigin
    src="https://unpkg.com/@onefootprint/footprint-js@latest/dist/footprint-js.umd.js"
  ></script>
</body>
```

or

```bash
# With NPM
npm install @onefootprint/footprint-js

# With yarn
yarn add @onefootprint/footprint-js
```

<inline-alert variant='info'>Make sure your application's permissions and content security policies are configured correctly to allow the Footprint integration. More info can be found [here](/integrate/content-security-policy).</inline-alert>

4. Render Footprint

In order to use our onboarding flows, you can either render a Footprint button, or use your own button and trigger Footprint when clicked.

### Vanilla HTML

If you would like to use a pre-built Footprint button, below is what your integration would look like for Vanilla HTML:

```html
<div data-footprint data-public-key="pb_test_VMooXd04EUlnu3AvMYKjMW"></div>
```

Make sure to add Footprint completion handlers on the `window` object:

```javascript
window.footprintCallbacks = {
  onComplete: function (token) {
    // TODO: User has finished the flow. This validation token can be used to see the fp_id of the
    // user, the auth method they used to log in, and their KYC status
  },
  onAuth: function (token) => {
    // User has authenticated. Optionally, this validation token can be used to see the fp_id
    // of the authenticated user and the auth method they used to log in
  },
};
```

Please refer to our guide for [Javascript](https://docs.onefootprint.com/sdks/javascript#verify-component-integration) for more details.

### React

If your app is built using React, you can use our React integration:

```typescript
import '@onefootprint/footprint-js/dist/footprint-js.css';
import { FootprintVerifyButton } from '@onefootprint/footprint-react';

const Page = () => {
  const handleCancel = () => {
    console.log('user canceled');
  };

  const handleComplete = (validationToken: string) => {
    console.log(validationToken);
  };

  return (
    <main>
      <FootprintVerifyButton
        publicKey="pb_test_VMooXd04EUlnu3AvMYKjMW"
        onComplete={handleComplete}
        onCancel={handleCancel}
      />
    </main>
  );
};
```

Please refer to our guide for [React](https://docs.onefootprint.com/sdks/react#footprint-verify-button-integration) for more details.

### Using your custom button

If you would like to use your own button and trigger Footprint, you can execute the following on click:

```javascript
const handleClick = () => {
  const component = footprint.init({
    kind: 'verify',
    publicKey,
    userData,
    onComplete: validationToken => {
      // TODO: Post the token to Footprint server to exchange with Footprint user ID
    },
  });
  component.render();
};
```

5. To customize the button placement, the easiest way is to wrap the button in a div and apply the styles to this div:

```html
<style>
  .container {
    display: flex;
    align-items: center;
    justify-content: center;
    height: 100vh;
  }
</style>

<div class="container">
  <div
    data-footprint
    data-kind="verify-button"
    data-props='{"publicKey": "pb_test_VMooXd04EUlnu3AvMYKjMW"}'
  ></div>
</div>
```

Click [here](https://github.com/onefootprint/examples/tree/master/idv/frontend-vanilla-js) to check out a full example.

## Verify the Footprint token server-side

### Setup and authentication

Get or create a Secret API Key on your [developer dashboard](https://dashboard.onefootprint.com/api-keys). The key will look something like `sk_test_CXUsbCR8j2kH6e5GeEl8eSBnQTIPCUaKpv`. Add this to the credential manager of your server-side app. To learn more about server-side API authentication, read the short guide on [API Authentication](/integrate/api-auth).

### Validate the onboarding token (from Step 5 above)

You should create an endpoint on your backend to handle signups. It should receive the validation token given in step 5 above from your frontend and pass it to Footprint’s backend in order to authenticate and verify the user using the <navigation-link fontFamily="code" href="/api-reference#post-onboarding-session-validate">POST /onboarding/session/validate</navigation-link> API:

```bash
curl -X POST https://api.onefootprint.com/onboarding/session/validate \
   -u sk_test_CXUsbCR8j2kH6e5GeEl8eSBnQTIPCUaKpv: \
   -d '{"validation_token": "vtok_udLaWUxPBo3fss603v8kY8k9ssjboxfwI"}'
```

Footprint will give you some information from this onboarding session. We’ll also give you a `fp_id`, which looks like `fp_id_GSxJr68GAf5jUT3pdL9ndjf7TLkA3GCX`.

```json
{
  "user": {
    "fp_id": "fp_id_GSxJr68GAf5jUT3pdL9ndjf7TLkA3GCX",
    "status": "pass",
    "requires_manual_review": false
  },
  ...
}
```

<inline-alert variant="info">
  Save the `fp_id` in your database tied to the individual user. It is the
  identifier you will send to request any user-specific information.
</inline-alert>

### Check the onboarding decision status

Using the `status` and `requires_manual_review` fields to decide whether or not to onboard this user to your product.

| Status    | What does this mean?                                                                                                                                                                                                                              |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pass`    | The rules defined on your playbook to verify the user evaluated to an outcome of `pass`.                                                                                                                                                                                     |
| `fail`    | The rules defined on your playbook evaluated to an outcome of `fail`.
| `none`    | Your playbook has no rules defined. |
| `pending` | Verifying this user's identity is taking longer than expected. This happens in rare cases where asynchronous actions are required to make a decision. A final decision will be delivered via [webhooks](/integrate/webhooks) to your application. |

| Review | What does this mean?                                                                              |
| ------ | ------------------------------------------------------------------------------------------------- |
| False  | We have made a decision automatically that does not require any manual review.                    |
| True   | Footprint suggests manually reviewing identified risks and manually checking acceptance criteria. |

### After onboarding

Once a user has onboarded onto Footprint, you can interact with the user via API using their `fp_id`. The most common action is to decrypt a user's information. For example:

```bash
curl -X POST https://api.onefootprint.com/users/fp_id_GSxJr68GAf5jUT3pdL9ndjf7TLkA3GCX/vault/decrypt \
   -u sk_test_CXUsbCR8j2kH6e5GeEl8eSBnQTIPCUaKpv: \
   -d '{"fields": ["id.dob", "id.last_name", "id.ssn4"], "reason": "getting started"}'
```

```json
{
  "id.dob": "1988-12-25",
  "id.last_name": "Doe",
  "id.ssn4": "1212"
}
```

You can learn more about the fields available for decryption [here](/vault/apis/#vault-fields).
